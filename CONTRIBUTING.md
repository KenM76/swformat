# Contributing to SWFormat

> Even though this is currently a solo project, this codifies the rules
> so future contributors (human or LLM agent) understand the discipline.

---

## The bar for any contribution

1. **Documentation-first.** The change must update or extend the
   relevant docs (`README.md`, `docs/ARCHITECTURE.md`, `docs/ROADMAP.md`,
   `docs/REVERSE_ENGINEERING.md`, `TESTING.md`) BEFORE the code is
   written. The "reconstruct from docs alone" bar applies.
2. **Diff-pair-backed.** Any new finding about the SOLIDWORKS file
   format must come with a documented diff-pair experiment showing how
   the finding was derived and falsified. See
   `docs/REVERSE_ENGINEERING.md` §1.
3. **Test before merge.** A new handler MUST have:
   - At least one synthetic corpus file exercising it
   (`tools/corpus_gen/<type>/<name>.csx`)
   - A golden manifest entry
   - A Layer 1 test asserting expected parse output
   - A Layer 2 round-trip test (read → write → byte compare under stable mask)
   - A Layer 3 SW-equivalence test (open the round-tripped file in SW
     via combridge; assert semantic equality)
4. **Tail bytes invariant.** Every byte read must be writeable. See
   `docs/ARCHITECTURE.md` §2 and `docs/REVERSE_ENGINEERING.md` §4.
5. **Lesson capture.** Every non-trivial finding produces a lesson at
   `lessons/lesson_<YYMMDD>_<slug>.md`. See
   `docs/REVERSE_ENGINEERING.md` §10 for the lesson template.

## Corpus IP rules

**STRICT.** Enforced by `.gitignore` and CI.

- All `*.sld*` files are blocked from commit by default.
- Only files generated by `tools/corpus_gen/` from .csx scripts may be
  whitelisted with an explicit negation in `.gitignore`.
- Real-world files (your own project work, client work, anything not
  synthetic) are referenced by absolute path in
  `test/corpus/corpus.config.json` with `license_status: owner`.
- A CI check fails the build if any `.sld*` file lands in
  `test/corpus/` without a corresponding `synthetic` entry in
  `corpus.config.json`.

This is non-negotiable. SWFormat must be IP-clean to ship as
open-source.

## Code style

- Python 3.11+ syntax
- `ruff check` clean (zero warnings)
- `mypy --strict` clean on `src/swformat/types.py` and Layer 1
  modules; permissive on M5+ stubs (per `pyproject.toml`)
- Line length 100
- Type hints required on all public functions
- Docstrings required on all modules and public functions
- **Comments must satisfy the dual-audience standard below.**

## Commenting standard (dual-audience) — HIGH PRIORITY

Documentation is a gate on every change, not an afterthought. **Every source
file must be commented for BOTH humans and LLMs.** Rationale: this is a
reverse-engineering project — "the documentation is the logic; the code is
the syntax that enacts it." Byte-level findings are hard-won and easily
lost; comments must let any reader (human or model) reconstruct the *why*
and the exact format from the file alone.

**Human-optimized comments** (prose, the "why"):
- Module docstring: purpose, place in the 5-layer model, the real problem it
  solves, an analogy where it clarifies.
- Function docstring: intent, the design choice and why over alternatives,
  gotchas, failure modes, a worked example where useful.
- Narrative that explains reasoning — never just restate the code.

**LLM-optimized comments** (structured, explicit, self-contained):
- Explicit **byte-offset / field-layout tables** for any binary structure
  (offset, size, type, meaning). Never force the reader to infer a layout.
- State **invariants and contracts** with labels: `INVARIANT:`,
  `PRECONDITION:`, `RETURNS:`, `RAISES:`.
- Use **canonical, consistent names** for format concepts so cross-file
  references resolve (e.g. CString = `FF FE FF <len> <utf16le>`, "object
  map", "class-ref tag").
- **Cross-reference the source of truth**: the hypothesis log
  (`research/empirical_findings/<topic>/log.md`) and the graduated lesson
  (`lessons/lesson_*.md`); cite exact byte evidence.
- Keep each file **self-contained** — enough context that a model reading
  only this file understands it.
- Record **provenance/limits**: which SW version verified, what's unconfirmed.

Applies to `src/swformat/**`, tests (state what the test pins down and why),
tools/CLIs (usage, behavior, exit codes), and generated `.csx`. When in
doubt, over-document — verbosity in docs is wanted here. Keep
`docs/ARCHITECTURE.md`, `docs/FORMAT_GUIDE.md`, `CHANGELOG.md`, and the
hypothesis logs updated on EVERY increment.

## Commit messages

```
<area>: <one-line summary>

Optional longer body explaining the WHY (not the WHAT — the diff
shows the what).

References:
- Lesson: lessons/lesson_YYMMDD_slug.md
- Diff pair: research/empirical_findings/<topic>/log.md
- Roadmap milestone: M2
```

Areas: `chunks`, `streams`, `carchive`, `io`, `api`, `tools`, `tests`,
`docs`, `corpus`, `meta`.

## How to propose a new milestone

1. Open a hypothesis log under `research/empirical_findings/<topic>/log.md`
2. Run 1-3 diff-pair experiments documenting what's learnable
3. Write the proposed milestone definition (pass criteria, effort,
   feasibility) as a draft in `docs/ROADMAP.md`
4. Adversarial review — challenge the effort estimate, the
   pass criteria, the prerequisites. Ideally do this via a workflow
   that runs separate research agents (see how M0-M6 were designed).
5. Accept the milestone into the roadmap with whatever modifications
   the adversarial review surfaced

## How to write a new stream handler (M2+)

The canonical sequence:

1. **Identify the stream by name.** Use `python -m swformat.tools.dump_streams`
   to inspect a known file; pick a stream that's currently `OpaqueStream`.
2. **Build a minimal diff-pair** isolating what changes in this stream.
3. **Implement `parse(data, doc_version)`** that returns structured data.
4. **Implement `serialize()`** that returns bytes.
5. **Add the handler to `src/swformat/streams/catalog.py`** mapping the
   stream name pattern to the handler class.
6. **Add tests:**
   - Layer 1: handler is correctly looked up by name
   - Layer 2: `parse(bytes) → serialize() == bytes` (the round-trip
     invariant)
   - Layer 3: write back into a Document, save, reopen in SW, verify
     the parsed data matches what SW sees
7. **Write the lesson.**

## When to NOT decode a stream

The `OpaqueStream` pass-through is a valid permanent disposition. Use
it for:

- Graphics caches that SW regenerates on every save (`Contents/DisplayLists`)
- Third-party-addin data (`ThirdPtyStore/*`) — opaque to us; preserve
  verbatim
- Save-history streams with timestamps (`_MO_VERSION_NNNNN/Biography`)
- Anything where the cost of decoding exceeds the value of modifying

The decision to leave a stream opaque is itself a documented finding.
Write a lesson noting WHY.

## Code review checklist

For self-review (or future external review):

- [ ] Docs updated before code
- [ ] Dual-audience comments present (human "why" + LLM byte-tables/
      invariants/cross-refs; see "Commenting standard")
- [ ] Tail-bytes invariant preserved (every byte read is writeable)
- [ ] New handler has Layer 1+2+3 tests
- [ ] New corpus files are synthetic (.csx generator under tools/corpus_gen)
- [ ] Lesson written and indexed
- [ ] CHANGELOG.md updated
- [ ] ROADMAP.md milestone status updated if applicable
- [ ] No `*.sld*` checked in outside the whitelist
- [ ] mypy clean on changed files
- [ ] ruff clean on changed files
- [ ] All tests pass

## Future contributors

Notes for external contributors:

1. Contributors will NOT have access to the maintainers' private
   `corpus.config.json` real-file entries. You'll need to either generate your
   own synthetic corpus via the corpus-generator scripts, or use your OWN
   legitimately-obtained real files for diff-pair experiments (see
   `docs/LEGAL.md`).
2. The Layer-3 (live-SOLIDWORKS) tests require a SOLIDWORKS license. CI runs
   Layers 1+2 (pure Python, no SW) only.
