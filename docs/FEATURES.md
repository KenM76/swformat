| **Sketch DIMENSIONS on real (assembly) drawings — counts length-PARAMETERS, not just display dims** | `read_sketch_dimensions` anchors on a 24-byte signature in the `moLengthParameter_c` parameter object and reports every such length-parameter's VALUE. This is EXACT on pure sketch-dimension drawings (synthetic fixtures, `two_dim`). **But on a real ASSEMBLY drawing it OVER-COUNTS display dimensions** — `moLengthParameter_c` is a generic length parameter used for display dimensions AND other embedded model length params, so the reader reports a SUPERSET. **SW-verified**: drw_60 has **106** view display dimensions (`GetDimensionCount2` × 46 views) but the reader reports **727** length-parameter objects. Distinguishing display dimensions needs the interned display-class (`moDisplayDistanceDim_c`/…) resolved via the CArchive class table — a follow-up. Also: KIND/refs/placement come from the display/handle strings, present only for the full-form instance (interned → `kind="unknown"`). |
| **CArchive round-trip model (M5.1 Step 1)** | `carchive.cusprops.roundtrip(data) == data` across the corpus: the `Contents/CusProps` **header** and **all user-record variants** (novalue / StringElem / PRP / MassProp) are re-serialized *structurally* (`reserialize_header`, `serialize_user_records`), byte-exact; only the cut-list container + close-time object-map dump tail is still carried verbatim (tail-bytes invariant). A no-orphan-bytes coverage gate (`check_user_list_coverage`) validates the user-list walk. No SW. |

The general object-map **ledger** (`carchive.objmap`) is built + validated — it round-trips the real `Contents/CusProps` header + all user records byte-exact and cross-checks against `_object_map_walk`'s threshold. Remaining: the object-map **insert/mutate** primitive (Step C) to replace the heuristic re-index + enable binary inserts — assessed marginal-value vs the working heuristic, so parked at a documented, resumable foundation. On assemblies/drawings a grow-beyond-span raises `SpanPreserveError` (relocate is rejected by asm/drw — they validate file size; needs the size-field decode). ## ❌ Out of scope by design (opaque pass-through)

These are read/copied verbatim but never interpreted — modifying them is not
a goal:

- `Contents/DisplayLists` — graphics cache (SW regenerates on save)
- `ThirdPtyStore/*` — third-party add-in data (may be proprietary)
- `_MO_VERSION_NNNNN/Biography` — save history (timestamps)
- Parasolid-backed geometry payloads inside CArchive

## Key limits to know

- **Verified on SOLIDWORKS 2026** (`_MO_VERSION_19000`). Older versions
  (2020/2024) and the pre-2015 OLE2 format are not yet covered.
- **Writes: span-preservation first; grow-beyond-span via relocate-to-EOF on PARTS.**
  An edit that fits the stream's ORIGINAL compressed size is written by padding
  back to that size (no offset shift — SW-verified, safe on ALL files: parts,
  assemblies, drawings). A grow that doesn't fit is written, **on PARTS**, by
  relocating the grown stream to the EOF and repointing its TOC record
  (SW-verified on v19000 parts — property add, config add, config rename-to-
  longer). On **assemblies and drawings** a grow-beyond-span raises
  `SpanPreserveError`: relocate is rejected (asm/drw validate file size, parts
  don't — root-caused 2026-06-11), and the old offset-shift fixup is **falsified**
  (it produces files SW rejects, `swFileRequiresRepairError`; retained only as an
  experimental opt-in, never a consumer path). Supporting asm/drw grows needs the
  obscured size-field decode (open).
- **Two save outputs of the same file are never byte-identical** (SW save is
  non-deterministic — even the length differs). So correctness is judged by
  *semantic equivalence on reopen*, not byte equality (see `TESTING.md`).
- **Layer-3 (live-SW) checks are manual** — they require a SOLIDWORKS
  session via combridge; the pure-Python layers (read / round-trip / TOC /
  property / CArchive read + round-trip / config rename / sheet rename /
  sketch-geometry read / relocate-grow) run without SW, on any OS, as the
  portable test suite (CI runs them on Linux). The real-file Layer-2 tests
  additionally run when a local corpus file is configured (else they skip).
